# Intent Ledger

This ledger preserves what the user actually asked for. Reviewers should use it to reject technically green work that misses the point.

| Source | Promise | Feature | Acceptance Proof |
|---|---|---|---|
| I want a really nice GUI for using llama.cpp so I dont have to memorize any commands or flags | Every llama.cpp flag is exposed as a widget — zero command-line memorization required | settings-flags | Every llama.cpp server flag has a corresponding widget in the Settings tab, and changing a widget changes the flag passed to the binary |
| I want it to have a very modern look and feel. Not some sort of dated windows XP sort of look. Professional and modern is what I want. | The UI must look professional and modern, using React/Tailwind components — no legacy widget aesthetics | settings-flags | The app uses modern React components with Tailwind CSS styling; visual review confirms no dated or legacy widget appearance |
| Nice looking widgets and dropdown menus. Nice and organized tab layout | The app has an organized tab layout with nice-looking widgets and dropdown menus | settings-flags | The main interface has clearly organized tabs (Server, Models, Settings, Fine-tuning, System Monitor) with modern widgets in each |
| Its for running and monitoring my local llm server. I want to easily load/unload models, fine tune, tweak settings, maximize performance of my hardware | Full control panel: server start/stop, model load/unload, settings tweak, fine-tuning, and hardware performance monitoring | server-control | All five tabs are functional and each maps to real llama.cpp functionality |
| including a tab to monitor system resources so you know exactly whats going on with your system | A System Monitor tab showing real-time CPU, RAM, and GPU metrics | system-monitor | System Monitor tab displays real-time CPU, RAM, and GPU metrics updating at least once per second |
| Already built — I have llama.cpp ready — Keeps the first version focused on the GUI; the app just points to your existing binary. | The app points to the user's existing llama.cpp binary — it does not download or build it | binary-config | First launch prompts for the binary path; the app never downloads or builds llama.cpp |
| Everything including fine-tuning — More ambitious first build; fine-tuning adds significant complexity but you get the full vision at once. | Fine-tuning is included in the first version and must work end-to-end | fine-tuning | Fine-tuning tab launches the finetune binary, training completes, and the output model file exists at the specified path |
| A downloadable package like a .deb or AppImage — Easy to install with a single file, shows up in your app menu like any normal application. | The app is distributed as a .deb or AppImage and appears in the Linux app menu | server-control | The app can be packaged as a .deb or AppImage and launched from the Linux application menu |
| I can do all the above AND fine-tune a model successfully — Requires the full pipeline including training to work, which is a higher bar. | The full pipeline including fine-tuning must actually work — not just look like it works | fine-tuning | Scenario test proves fine-tuning produces a real output model file |
| Show a clear error message and let you fix it and try again — Keeps you in control; the app explains what went wrong in plain language so you can adjust and retry. | Errors are shown as clear in-app messages in plain language, and the user can fix and retry | error-handling | Every failure condition produces a visible in-app error message with a plain-language explanation and a retry option |
| Scenario tests for the critical workflows - prove the most important user paths with pass/fail evidence. | Critical workflows are proven by scenario tests with pass/fail evidence | server-control | Each feature has scenario tests defined and passing that prove the critical user paths |
| Preserve current user workflows and data - new work should not break existing launch, save, or run paths. | Existing user workflows and data must not be broken by new work | binary-config | Config file format changes are backward-compatible; existing settings are preserved across updates |
| Finish missing real behavior - replace placeholders and incomplete flows with working code. | No placeholder buttons or incomplete flows — every control maps to real behavior | settings-flags | Every visible control is wired to real llama.cpp functionality — no stubs or placeholders |
| Tool permissions, memory, and confirmation rules - keeps autonomous behavior predictable and safe. | Agent operates with locked tool permissions, memory rules, and confirmation rules | binary-config | Agent rules are defined in conventions: no direct file system access from frontend, no modifying llama.cpp binary, confirm before changing config format |
